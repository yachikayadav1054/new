1. Introduction to Power Pages

What is Power Pages?

  • • Low-code platform for building external-facing websites
  • • Connected directly to Microsoft Dataverse
  • • Enables secure data access for external users (customers, partners, vendors)
  • • Part of Microsoft Power Platform ecosystem

Architecture Overview

Power Pages Website (Frontend)
↓
Authentication Layer
↓
Web Roles & Table Permissions
↓
Microsoft Dataverse (Backend)

Power Pages vs Power Apps Portal

Key Point: Power Apps Portal is the OLD name. Power Pages is the NEW branding (same technology, rebranded in 2022).

  • • Same licensing model (per authenticated user or capacity)
  • • Modern design studio in Power Pages
  • • Legacy portal management app still available
{/* Section 2 */}

2. Core Components

Websites

  • • Container for all portal content
  • • Each website has unique Website ID and binding
  • • Multiple websites possible per environment

Tables & Forms

Basic Forms:

  • • Single record CRUD operations
  • • Modes: Insert, Edit, ReadOnly
  • • Configuration: Success redirect, validation, file uploads

Advanced Forms:

  • • Multi-step wizard-style data collection
  • • Conditional navigation between steps
  • • Example: Loan application (Personal → Financial → Documents)

Lists

  • • Grid view of Dataverse records
  • • Based on Dataverse views (system or custom)
  • • Features: Search, filter, pagination, actions (Create/Edit/Delete)
  • • Respects table permissions (automatic security)

Web Roles

  • • Security containers grouping users
  • • Define WHO can access what
  • • Assigned via N:N relationship with Contact
  • • Types: Authenticated Users, Anonymous Users, Custom Roles
  • • One user can have multiple roles (cumulative permissions)

Table Permissions (Critical)

5 Permission Scopes:

  • 1. Global: All records in table (Admin access)
  • 2. Contact: Records where user's contact = lookup field
  • 3. Account: Records where user's account = account lookup
  • 4. Parent: Child records based on parent record access
  • 5. Self: Only user's own contact record

Privilege Types: Create, Read, Write, Delete, Append, Append To

Authentication Providers

  • • Azure AD (Entra ID) - Enterprise SSO
  • • Azure AD B2C - Social + enterprise identities
  • • Local Authentication - Username/password in Dataverse
  • • OAuth 2.0 - Google, Facebook, LinkedIn, Microsoft
  • • SAML 2.0 - Custom enterprise providers

Site Settings

  • • Key-value pairs for configuration
  • • Control behavior without code changes
  • • Examples: Authentication/Registration/Enabled, Search/Enabled

Content Snippets

  • • Reusable text blocks across pages
  • • Support HTML and Liquid syntax
  • • Editable by non-technical content authors

3. Security (Interview Critical)

⚠️ Most Important Section for Interviews

Deep understanding of security is essential for Solution Architect role

Table Permission Examples

Example 1: Contact Scope (Most Common)

Scenario: User sees only THEIR cases

  • • Table: Case
  • • Scope: Contact
  • • Relationship: customerid_contact
  • • Privileges: Read ✓, Write ✓, Delete ✗
  • • Result: User sees cases where customerid = their Contact ID

Example 2: Account Scope

Scenario: User sees ALL cases for their company

  • • Table: Case
  • • Scope: Account
  • • Relationship: customerid_account → Contact.parentcustomerid
  • • Result: All users from same company see shared cases

Example 3: Parent Scope

Scenario: User can add notes only on accessible cases

  • • Table: Annotation (Notes)
  • • Scope: Parent
  • • Parent Relationship: Case (objectid_incident)
  • • Privileges: Read ✓, Create ✓
  • • Result: Notes visible only on cases user can already access

Record-Level Security Flow

  1. 1. User requests a Case record (ID = 123)
  2. 2. Portal checks: Is user authenticated? → Yes
  3. 3. Retrieves user's Web Roles → ["Customer", "Premium"]
  4. 4. Gets Table Permissions for Case + these roles
  5. 5. Evaluates scope conditions (Contact/Account/Parent)
  6. 6. Applies filter to Dataverse query
  7. 7. If record matches → Display | If not → Access Denied
  8. 8. Security enforced server-side (cannot be bypassed)

Real-World Security Scenario

Multi-Tier Vendor Portal

5. Integration Approaches

Three Ways to Call External APIs

Approach 1: Power Automate (Recommended)

  • • Best For: Asynchronous operations, simple integrations
  • • Flow: Form submit → Dataverse record → Power Automate triggers → Call API → Update record
  • • Example: Submit support ticket → Create in ServiceNow
  • • Pros: No code, built-in connectors, easy maintenance
  • • Cons: Not real-time (1-2 min delay)

Approach 2: Azure Functions

  • • Best For: Real-time response, complex logic
  • • Flow: JavaScript → HTTPS POST → Azure Function → External API → Response to portal
  • • Example: Credit score check during registration
  • • Pros: Real-time, full control, high performance
  • • Cons: Requires coding, additional Azure resource

Approach 3: Dataverse Plugin

  • • Best For: Complex server-side business logic
  • • Flow: Form submit → Record created → Plugin executes → API call → Update record
  • • Example: Auto-create ticket in external system on case creation
  • • Pros: Server-side, secure, no client exposure
  • • Cons: .NET development required

Common Integration Patterns

  • • Email Notifications: Power Automate triggered on status change
  • • Scheduled Sync: Recurrence trigger to sync external data daily
  • • Document Storage: SharePoint integration for large files
  • • Real-time Validation: Azure Function for address/credit checks

Securing API Calls

OAuth 2.0 Best Practices:

  • • Use client credentials flow for service-to-service
  • • Store credentials in Azure Key Vault (never in code)
  • • Implement in Power Automate or Azure Function
  • • Never expose API keys in JavaScript
  • • Use managed identities where possible

6. ALM & Deployment

Environment Strategy

  • • DEV: Development environment (individual dev sandboxes)
  • • TEST: Integration testing
  • • UAT: User acceptance testing
  • • PROD: Production (live environment)

Solution Components

Included in Solutions:

  • • Website record, Web pages, Web templates
  • • Lists, Forms configurations
  • • Table permissions, Web roles
  • • Content snippets

NOT in Solutions:

  • • Actual content (blog posts, KB articles)
  • • Domain bindings
  • • Some site settings (manual migration needed)

Deployment Process

  1. 1. Develop in DEV environment
  2. 2. Add components to solution
  3. 3. Export as managed solution
  4. 4. Import to TEST → Validate
  5. 5. Import to UAT → User testing
  6. 6. Import to PROD → Go live
  7. 7. Clear portal cache (restart portal)
  8. 8. Verify table permissions and web roles

Deployment Checklist

Pre-Deployment:

  • ☐ Run solution checker
  • ☐ Backup target environment
  • ☐ Test in lower environment
  • ☐ Document changes

Post-Deployment:

  • ☐ Clear portal cache
  • ☐ Test critical pages
  • ☐ Verify security
  • ☐ Monitor error logs

Version Control Best Practices

  • • Export and unpack solutions for version control
  • • Use Git branches: main, develop, feature/*, bugfix/*
  • • Never commit API keys or credentials
  • • Mandatory PR reviews for main branch
  • • Tag releases (v1.0, v1.1, etc.)

7. Top 20 Interview Questions

{/* Q1-10 */}

Q1: What are the authentication providers in Power Pages?

Answer: Azure AD (Entra ID), Azure AD B2C, Local Authentication, OAuth 2.0 (Google, Facebook, LinkedIn), SAML 2.0 for custom enterprise providers.

Q2: Difference between Web Roles and Table Permissions?

Answer: Web Roles define WHO (groups like Customer, Partner). Table Permissions define WHAT data each role can access. Web Roles are assigned to users; Table Permissions are assigned to Web Roles. One user can have multiple roles with cumulative permissions.

Q3: What are the 5 Table Permission scopes?

Answer: 1. Global (all records), 2. Contact (user's contact = lookup), 3. Account (user's account = lookup), 4. Parent (child records based on parent), 5. Self (only own contact record)

Q4: How does record-level security work?

Answer: User requests record → Portal checks authentication → Gets user's Web Roles → Retrieves Table Permissions → Evaluates scope → Applies filter to Dataverse query → If match, display; else access denied. Server-side enforcement.

Q5: What is Liquid and its use?

Answer: Server-side template language for dynamic content. Used for displaying user data, conditional rendering, loops, date formatting. Processes before HTML rendering.

Q6: Three ways to call external APIs?

Answer: 1. Power Automate (async, easy), 2. Azure Functions (real-time), 3. Plugin (server-side, C#). Use Power Automate for simple, Azure Functions for real-time needs.

Q7: Basic Forms vs Advanced Forms?

Answer: Basic = Single-step CRUD. Advanced = Multi-step wizard with conditional navigation. Use Advanced for complex data collection.

Q8: How to deploy from DEV to PROD?

Answer: Add to solution → Export as managed → Import to TEST → UAT → PROD → Clear cache → Verify security. Some site settings need manual migration.

Q9: What are Content Snippets?

Answer: Reusable text blocks stored in Dataverse. Support HTML and Liquid. Used for headers, footers, messages. Editable by non-technical users.

Q10: How to secure API calls?

Answer: Use OAuth 2.0 client credentials flow. Store credentials in Key Vault. Implement in Power Automate or Azure Function. Never expose in JavaScript. Use managed identities.

{/* Q11-20 */}

Q11: User sees only THEIR cases - how?

Answer: Table Permission: Table=Case, Scope=Contact, Relationship=customerid_contact, Privileges=Read/Write. User sees cases where customerid = their Contact ID.

Q12: Company users see all company cases - how?

Answer: Use Account scope. Relationship: customerid_account → Contact.parentcustomerid. All users with same parent account see shared cases.

Q13: Real-time address validation - how?

Answer: Azure Function with address API → JavaScript calls on field blur → Function validates → Return result → Show error/success → Block if invalid.

Q14: Immediate email on form submit - how?

Answer: Power Automate (1-2 min delay) or Azure Function (instant via SendGrid).

Q15: Secure file uploads - how?

Answer: Enable on form → Configure max size → Table Permissions → Validate file types → Virus scan via Azure Function. Files in Dataverse or Azure Blob.

Q16: Different API URLs per environment - how?

Answer: Site Settings (ExternalAPI/BaseURL, different per env) or Environment Variables in Power Automate.

Q17: Search across multiple tables - how?

Answer: Built-in portal search (enable in settings) or custom search (JavaScript → Web API → FetchXML → Display results).

Q18: Manage frequent role changes - how?

Answer: Web Roles in Dataverse → Assign via N:N at runtime → Power Automate auto-assign based on criteria → No deployment needed.

Q19: Portal slow - how to optimize?

Answer: Enable output caching → Optimize Liquid queries → Pagination → Compress images → CDN for static files → Entity caching → Optimize views.

Q20: Multi-language support - how?

Answer: Enable portal languages → Content snippets per language (welcome-en, welcome-es) → Check user.language in Liquid → Display appropriate snippet.
