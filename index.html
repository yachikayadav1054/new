Power Pages Fundamentals

▼
What is Power Pages?
A low-code platform to build secure, external-facing business websites (portals) connected to Microsoft Dataverse. It is part of Power Platform. External users (customers, partners, citizens) can interact with business data through forms, lists, and pages — without needing a Dataverse/M365 license.
Power Pages was rebranded from Power Apps Portals in 2022. Knowing this transition shows awareness. The underlying technology is the same but the licensing, maker experience, and branding changed.
Key use cases for Power Pages
  • Customer self-service portals (case submission, tracking)
  • Partner portals (deal registration, lead management)
  • Employee self-service (HR portals, onboarding)
  • Community portals (forums, knowledge base)
  • Government/citizen service portals
  • Event registration and survey sites
How does Power Pages differ from Canvas Apps or Model-Driven Apps?
FeaturePower PagesCanvas AppModel-Driven App
UsersExternal (anonymous/authenticated)Internal (licensed)Internal (licensed)
UIWebsite (HTML/CSS)App screensData-model driven
AccessBrowser URL, no app installApp/browserBrowser
AuthAAD, B2C, OAuth, LocalAAD onlyAAD only
LicensePer authenticated user / capacityPer user/appPer user/app
Power Pages Licensing Model
  • Authenticated users: Licensed per login (monthly active user) or per site capacity
  • Anonymous users: Included but limited to read-only public pages
  • Maker: Requires Power Pages maker license (or Power Apps premium)
  • Included free trial capacity per environment for testing
Anonymous users are NOT free — they count against capacity limits if they're accessing Dataverse-connected pages. Truly static pages don't consume capacity, but any Dataverse read does.

What is the Portal Base solution?
A managed solution installed in the Dataverse environment when a Power Pages site is provisioned. It contains all portal-specific tables (adx_*), security roles, and configurations. You customize ON TOP of this, not inside it.

Pages, Web Templates & Layouts

▼
Web Page hierarchy
Web pages form a tree structure. Each page has: a Parent Page, a partial URL (slug), and is associated with a Page Template. The full URL = parent URL + partial URL. The root page (Home) has partial URL /.
Web Page vs Content Page (Localization)
Each Web Page has one root record (adx_webpage with isroot=true) and one or more content/localized records per language. The content page holds the actual HTML/Liquid content. The root page holds the URL and navigation metadata.
When editing page content, you're editing the content/localized record — NOT the root record. Confusing these two causes "changes not showing" issues. Classic trap!
What is a Page Template?
Links a Web Page to a Web Template (or a rewrite path). Types:
  • Web Template: Uses a Liquid-based web template for rendering
  • URL Rewrite: Redirects to an ASPX page (used for system pages like login, profile)
What is a Web Template?
A Dataverse record containing Liquid + HTML + CSS + JS that defines how a page is rendered. Web Templates can include other templates using {% include 'template-name' %}. This is the core rendering engine of Power Pages.
What are Content Snippets?
Reusable, editable text or HTML blocks stored in Dataverse. Referenced in templates using {% editable snippets 'Snippet Name' %}. Supports multilingual content. Allows non-technical users to edit copy without touching templates.
A "Footer Copyright Text" content snippet can be edited by business users in the Power Pages Studio without touching any code.

Authentication & Identity

▼
Authentication options in Power Pages
ProviderNotes
Azure AD (Entra ID)Internal users / employee portals
Azure AD B2CCustomer-facing; supports social login (Google, Facebook)
OAuth 2.0 / OpenID ConnectAny OIDC-compliant provider (Okta, Ping, Salesforce)
SAML 2.0Enterprise SSO (ADFS, etc.)
Local AuthenticationUsername/password stored in Dataverse (discouraged for production)
AnonymousNo login; public pages only
For B2C scenarios (customer portals), Azure AD B2C is the go-to. It handles social login, MFA, and custom user flows. Mentioning this shows real-world experience.
How does user authentication map to Dataverse?
When a user logs in, Power Pages creates or links a Contact record in Dataverse. The Contact is the authenticated user's identity. Their web roles, table permissions, and data access are all tied to this Contact record.
Authenticated users in Power Pages are Contacts — NOT Dataverse users (systemusers). They don't have security roles in the traditional sense; they have Web Roles. This is a very common interview trap!
What is the Portal Contact record?
A standard Dataverse Contact record associated with a portal user via the adx_portalcontact relationship. Contains web authentication details. When a user registers, a Contact + portal login record is created.
What is Invitation Redemption?
A mechanism to invite specific contacts to register on the portal. An Invitation record is created for a Contact, generating a unique invite code. When redeemed, the contact is registered and assigned specified web roles automatically.
Multi-factor Authentication (MFA) in Power Pages
MFA is enforced via the identity provider (Azure AD / B2C / OIDC). Power Pages itself doesn't enforce MFA — it delegates to the IdP. Configure MFA policies in Azure AD B2C user flows or Conditional Access policies.

Table Permissions

▼
What are Table Permissions?
Records in Dataverse (adx_entitypermission) that define what Dataverse table data external portal users can access. Without table permissions, users get no data — all data access is denied by default.
By default, NO data is accessible to portal users. Table Permissions must be explicitly configured. If a form or list shows no data, missing table permissions is the #1 cause. Interviewers love this!
Table Permission Scopes
ScopeDescriptionUse Case
GlobalAccess ALL records in the tableRead-only reference data (countries, categories)
ContactOnly records where a lookup points to the logged-in ContactUser's own cases, orders
AccountRecords related to the user's parent AccountAll cases for the user's company
SelfThe Contact record itselfUser profile page
ParentRecords related via a parent table permission (child)Line items under an order the user can access
Scope = Contact: A portal user can only read/update Cases where Case.CustomerID = logged-in Contact's ID. They can't see other users' cases.
Table Permission Privileges
Read, Create, Write (Update), Delete, Append, Append To — same as Dataverse security roles but applied to external portal users via web roles.
How are Table Permissions assigned?
Table Permissions are linked to Web Roles. When a Contact has a web role, they inherit all table permissions assigned to that web role. Multiple web roles = union of all permissions.
Parent Scope — how does it work?
Used for child records. Define a parent table permission first (e.g., Orders with Contact scope). Then create a child table permission for Order Lines with scope = Parent, referencing the Orders permission. Users can access order lines only for orders they can access.
Parent scope is the most powerful and most complex. Drawing a hierarchy diagram (Order → Order Lines) in interviews demonstrates deep understanding.

Page Permissions & Web Roles

▼
What are Web Roles?
Groups assigned to authenticated contacts to control: page visibility, table permissions, and site features. Every authenticated user gets the Authenticated Users web role. Anonymous users get the Anonymous Users web role automatically.
Page Permissions (Web Page Access Control Rules)
Control which web roles can view or manage a web page. Two rights:
  • Grant Change: Allows editing page content (for front-side editing)
  • Restrict Read: Hides the page from users not in the specified web role
If no Restrict Read rule exists, page is visible to everyone (including anonymous).
Page permissions are NOT inherited by child pages by default. You must explicitly add access control rules to child pages OR check "Apply to child pages." Missing this causes child pages to remain public!
A "My Profile" page has Restrict Read for "Authenticated Users" role — only logged-in users can see it. Anonymous users hitting the URL are redirected to login.
Authenticated Users vs Anonymous Users roles
  • Authenticated Users: Auto-assigned to all logged-in contacts. Base role for all users.
  • Anonymous Users: Auto-assigned to non-logged-in visitors. Use for public page table permissions.
You cannot delete these system roles.

Forms (Basic Forms & Multistep Forms)

▼
Types of Forms in Power Pages
TypeOld NameDescription
Basic FormEntity FormSingle-step form mapped to a Dataverse form; Create/Edit/ReadOnly modes
Multistep FormWeb FormWizard-style multi-page form across multiple steps/tables
Basic Form — key configuration
  • Table Name: Which Dataverse table
  • Form Name: Which Dataverse form layout to use
  • Mode: Insert (create), Edit (update), ReadOnly
  • Record Source Type: Query String (pass ID in URL), Current Portal User (Contact), Fixed Record
  • On Success: Redirect, show message, or stay on page
  • Enable Table Permissions: Must be checked to enforce security
"Enable Table Permissions" on a Basic Form defaults to OFF in legacy portals. If it's unchecked, ANY authenticated user can edit ANY record — a major security risk. Always enable it!
Multistep Form (Web Form) structure
Multistep Form → Steps → each Step = Basic Form config
Each step can target different tables, modes, and conditions. Steps support: Condition (branch), Redirect, Load Form, Load Tab.
Insurance Claim wizard: Step 1 = Claimant details (Contact), Step 2 = Claim details (Case), Step 3 = Upload documents (Notes), Step 4 = Confirmation.
Form Events — where to add custom JS?
Basic Forms support:
  • Custom JavaScript: Added directly in the form's JS tab or via web resource
  • Use $(document).ready() or the portal's event model
  • addEventListener("loaded", ...) for Multistep Forms
  • Field IDs follow pattern: _fieldschemaname_* — use $("#fieldname") with jQuery
Form Metadata — what is it?
Extra configuration records linked to a Basic Form step that modify field behavior: change label, add validation, set default values, mark required, hide fields — without modifying the underlying Dataverse form.
Form Metadata is very powerful — you can have one Dataverse form but multiple portal form configurations. Useful when the same table is used in different portal contexts with different field requirements.

Lists (Entity Lists / Table Lists)

▼
What is a List (formerly Entity List)?
A component that renders a Dataverse View as a grid/list on a Power Pages page. Supports search, filter, sorting, pagination, and actions (open record, create new, run workflow/action).
Key List configuration options
  • Table & View: Which table and which Dataverse view to display
  • Page Size: Records per page (default 10)
  • Search: Enable/disable search bar
  • Filters: Column-based OData filters, user-configurable filters
  • Actions: Create button (opens Basic Form), View (detail page), Edit, Delete
  • Enable Table Permissions: Must be ON to enforce row-level security
  • Override Column Attributes: Rename columns, show/hide without changing the view
Just like Basic Forms, Lists have an "Enable Entity/Table Permissions" toggle. If OFF, all authenticated users see ALL records regardless of scope. Always turn this ON in production!
List Actions — what can they do?
ActionDescription
CreateOpens a Basic Form for new record creation
View / Edit / DeleteRow-level actions with confirmation
DownloadExport list to Excel
Workflow / ActionTrigger a Power Automate flow or Custom Action on selected record
OData Filters on Lists
Lists support OData filter expressions to pre-filter records. Can be static (hardcoded) or dynamic using Liquid expressions (e.g., filter by current user's account ID using {{ user.parentcustomerid.id }}).

Liquid Templating

▼
What is Liquid in Power Pages?
An open-source templating language (by Shopify) extended by Microsoft for Power Pages. Used inside Web Templates, Content Snippets, and page content to dynamically render Dataverse data, handle logic, and control page output.
Liquid Syntax Basics
SyntaxPurposeExample
{{ }}Output expression{{ user.fullname }}
{% %}Logic/tag{% if user %} ... {% endif %}
{# #}Comment{# This is a comment #}
Key Liquid Objects in Power Pages
ObjectDescription
userCurrently logged-in Contact record
pageCurrent web page record
websiteCurrent website record
requestHTTP request (URL, params, cookies)
sitemapSite navigation tree
snippetsContent snippets collection
settingsSite Settings key-value pairs
entitiesQuery Dataverse records (respects table permissions)
{% assign cases = entities['incident'] | where: 'customerid', user.id %}
{% for case in cases %} {{ case.title }} {% endfor %}
The entities object respects table permissions automatically. If no permission is set, entities returns nothing — great for security. Always use it instead of raw fetchxml unless necessary.
FetchXML in Liquid
Use {% fetchxml %} tag for complex queries (joins, aggregates, ordering) that the entities shorthand can't handle.

{% fetchxml result %}
<fetch><entity name="account"><attribute name="name"/></entity></fetch>
{% endfetchxml %}
{% for a in result.results.entities %} {{ a.name }} {% endfor %}
Liquid includes and block/extends pattern
  • {% include 'template-name' %} — include a reusable template fragment
  • {% extends 'base-template' %} + {% block content %}{% endblock %} — page inheritance pattern for consistent layout
Liquid runs SERVER-SIDE — it cannot react to user interactions post-page-load. For dynamic client-side behavior after page load, you need JavaScript. Don't mix up Liquid (server render) with JS (client-side).

JavaScript & CSS Customization

▼
How to add custom JS/CSS to Power Pages?
  • Web Files: Upload JS/CSS files as Web Files in Dataverse; reference via URL
  • Web Template: Embed directly in template source
  • Basic/Multistep Form JS tab: Form-specific JS added directly in portal config
  • Content Snippets: Small inline scripts/styles
  • Power Pages Studio → Custom Code: Site-wide header/footer injection
jQuery availability in Power Pages
jQuery is included by default in Power Pages. Safe to use $(document).ready() patterns. Portal also includes Bootstrap for responsive layout.
Form field naming convention for JS
Fields on portal forms are rendered as: id="<tableschemaname>_<fieldschemaname>" but the exact ID can vary. Use browser dev tools to inspect. Best practice: use $("[id$='_fieldname']") to match end of ID for robustness.
How to validate form fields with JS?
Power Pages uses jQuery Validate plugin. Add custom validation rules using: Page_Validators array (legacy) or custom validator functions. For blocking save: add to Page_Validators with validationFunction that returns true/false.
Power Pages forms use ASP.NET WebForms validation under the hood. Page_Validators is the Microsoft-documented way to add client-side validation that integrates with the form's validation summary.
Bootstrap version in Power Pages
Legacy portals use Bootstrap 3. New Power Pages sites use Bootstrap 5. This matters for CSS class names (e.g., col-md-6 vs flexbox utilities). Always confirm which version is in use before writing custom CSS.

Power Pages Web API

▼
What is the Power Pages Web API?
A RESTful API endpoint exposed by Power Pages that allows client-side JavaScript to perform CRUD operations on Dataverse tables directly from the browser — without calling the Dataverse Web API directly. It respects table permissions and web roles automatically.
Web API endpoint format
/_api/<tablename> — e.g., /_api/contacts
Supports OData query options: $select, $filter, $orderby, $top, $expand.
How to enable the Web API?
  1. Enable via Site Setting: Webapi/<tablename>/enabled = true
  2. Set which fields are accessible: Webapi/<tablename>/fields = comma-separated field names or *
  3. Ensure Table Permissions are configured for the web role
Web API is DISABLED by default for every table. Forgetting the Site Setting is the #1 cause of 403 errors when using the portal Web API. Must explicitly opt-in per table!
Web API — CSRF protection
POST/PATCH/DELETE requests require an anti-CSRF token in the request header: __RequestVerificationToken. Retrieve it from the hidden input on the page: $("input[name='__RequestVerificationToken']").val().
GET: fetch('/_api/incidents?$select=title,statuscode&$filter=statecode eq 0')
POST: include __RequestVerificationToken header + JSON body
The Portal Web API is scoped to the logged-in Contact's permissions. It's safer than calling Dataverse Web API directly (which would need a service account or OAuth token exposed to the client).

Search & Content Management

▼
Search in Power Pages
Power Pages includes a built-in Lucene-based search index (or Azure Cognitive Search in newer versions) that indexes: web pages, knowledge articles, forums, and configured Dataverse table records. Search respects security — users only see results they have table permissions for.
How to enable Dataverse table search?
  1. Enable Site Setting: Search/Enabled = true
  2. Create a Search Boost or configure the search view per table
  3. Set the table's "Quick Find View" in Dataverse
  4. Re-index via portal admin (/_services/about page)
Knowledge Base integration
Power Pages has native integration with Dynamics 365 Knowledge Articles. Configure via KnowledgeManagement site settings. Articles are published to portal and searchable. Supports ratings, feedback, and filtering by categories/products.
Forums in Power Pages
Community portal template includes Forum functionality: forum threads, posts, voting, subscriptions. Stored in Dataverse portal tables (adx_communityforum). Controlled by table permissions and web roles.
Multilingual Support
Power Pages supports multiple languages via Portal Languages. Each web page has a content record per language. Users can switch language via a language picker. Content Snippets also support language variants. Configure via adx_portallanguage records.

Dataverse Integration

▼
How does Power Pages connect to Dataverse?
Power Pages runs in the same environment as Dataverse. It connects using a portal application user (a special system user) with specific privileges. All portal operations go through this app user, governed by table permissions for external users.
Portal Application User
An auto-created Dataverse system user named "Power Pages [site name]". It has the System Administrator role by default (in some versions) or a scoped portal role. This is how Power Pages accesses Dataverse server-side. Do NOT confuse with portal Contacts (external users).
The portal Application User runs with elevated privileges internally. Table permissions are what restrict EXTERNAL user access — not the app user's security role. Security is enforced at the portal layer, not at the app user level.
Notes & Attachments on forms
Enable "Enable Notes" on a Basic Form to allow file uploads stored as Dataverse Notes (annotations). Configure: allowed file types, max size, view/delete permissions. Files stored in Dataverse blob storage.
SharePoint integration for file storage
If SharePoint integration is enabled on the Dataverse environment, portal file uploads can be stored in SharePoint document libraries instead of Dataverse notes. Configure via SharePoint/Enabled site setting and SharePoint document location setup.
Triggering Power Automate from Portal
  • Via Basic Form On Save action: trigger a workflow/flow on record create/update
  • Via List Action button: run flow on selected record
  • Via Portal Web API + client-side logic: custom HTTP trigger flows
  • Dataverse plugin on table: fires when portal creates/updates records

Power Automate & Plugins with Power Pages

▼
When do Dataverse Plugins fire for portal operations?
Plugins registered on standard Dataverse messages (Create, Update, Delete) fire normally when portal forms submit. The portal app user is the InitiatingUserId unless impersonation is configured. Plugins don't know the request came from portal vs. API — they just see a Dataverse operation.
How to identify portal-initiated operations in plugins?
Check context.InitiatingUserId against the portal application user's ID. Or check for a custom field/flag set by the portal form before the plugin uses it as a discriminator.
Sending emails from portal actions
Options:
  • Power Automate flow triggered on record create (most common)
  • Dataverse plugin using SendEmail message
  • Portal's built-in email confirmation (for invitation redemption, registration)
  • Custom SMTP via Power Automate connector
Custom Actions (APIs) called from portal
Portal Liquid can call Dataverse Custom Actions using: {% entityform ... %} with action configuration, or via Portal Web API JavaScript: POST to /_api/<tablename>(<id>)/Microsoft.Dynamics.CRM.<ActionName>.

Site Settings & Configuration

▼
What are Site Settings?
Key-value configuration records (adx_sitesetting) stored in Dataverse that control portal behaviour. Changed via Portal Management App or Design Studio → Set up. No code deployment needed — changes take effect after portal cache clear.
Critical Site Settings to know
SettingPurpose
Authentication/Registration/EnabledAllow new user registration
Authentication/Registration/LocalLoginEnabledEnable local username/password login
Webapi/<table>/enabledEnable Web API for a table
Webapi/<table>/fieldsFields accessible via Web API
Search/EnabledEnable portal search
HTTP/X-Frame-OptionsClickjacking protection (SAMEORIGIN/DENY)
Site/EnableDefaultHtmlEncodingXSS protection (encode HTML output)
KnowledgeManagement/DisplayNotesShow KB article notes on portal
Portal Cache and how to clear it
Power Pages caches configuration (web templates, site settings, permissions) for performance. After changes in PMA, clear cache via:
  • Admin Center → Portal Actions → Clear Cache
  • Append /_services/about to URL → "Clear Cache" button (for admins)
Without clearing cache, changes may not reflect for up to 15 minutes.
Making changes in Portal Management App and not seeing them? Clear the portal cache. Forgetting to clear cache is the most common "why isn't it working" scenario in portal development!

File & Notes Attachments

▼
How do file uploads work in Power Pages?
Basic Forms with "Notes" enabled store uploaded files as Dataverse Annotation (Note) records linked to the parent record. Each file = one annotation with the document body stored as base64 in Dataverse (or SharePoint if integrated).
File attachment configuration options
  • Allowed MIME types (restrict file types)
  • Maximum file size (default: 10 MB, configurable via site setting)
  • Restrict note to table (prevent users from seeing unrelated notes)
  • Allow multiple file uploads per form submission
Site setting for max file upload size
SharePoint/MaxUploadSize or the Dataverse email attachment size setting. For Dataverse notes: the attachment size limit is set in Dataverse System Settings (Email tab → "Maximum file size").
Web Files — what are they?
Static assets (CSS, JS, images, PDFs) stored as Dataverse Web File records (adx_webfile). Accessible via URL path. Used to serve custom stylesheets, scripts, and downloadable documents directly from the portal.
Web Files have their own page permissions — you can restrict download of a PDF to authenticated users only by setting a Web Page Access Control Rule on the parent Web File page.

ALM, Solutions & Deployment

▼
Is Power Pages content in Dataverse solutions?
Yes — all portal configuration (web pages, templates, forms, lists, site settings, permissions) is stored as Dataverse records in portal tables (adx_*). These can be included in Dataverse solutions for ALM just like any other component.
How to migrate a Power Pages site between environments?
  1. Option A — Solution export/import: Add portal components to a solution; export unmanaged; import to target
  2. Option B — Configuration Migration Tool (CMT): Export portal configuration data as a ZIP; import to target environment. Best for data-heavy migrations.
  3. Option C — PAC CLI: pac paportal download/upload commands for source-controlled portal deployments
PAC CLI for Power Pages
CommandPurpose
pac paportal listList portals in environment
pac paportal download --path ./portalDownload portal content to local folder
pac paportal upload --path ./portalUpload local portal content to environment
PAC CLI portal download stores web templates, pages, content snippets as YAML/HTML files — perfect for Git-based version control. This is the modern DevOps approach for Power Pages.
Common deployment gotchas
  • Web roles must exist in target environment (create them or include in solution)
  • Table permissions reference web roles — deploy roles before permissions
  • Authentication provider settings are environment-specific (client IDs, secrets) — update site settings after deploy
  • Website binding must be reconfigured in target environment

Performance, Security & Best Practices

▼
Top Security Best Practices
  • Always enable Table Permissions on Lists and Basic Forms
  • Use most restrictive scope possible (Contact > Account > Global)
  • Disable local authentication in production; use AAD/B2C
  • Set HTTP/X-Frame-Options = SAMEORIGIN to prevent clickjacking
  • Set Site/EnableDefaultHtmlEncoding = true to prevent XSS via Liquid
  • Never expose sensitive fields in Web API — be explicit with fields setting
  • Use Column Security Profiles in Dataverse to restrict sensitive fields
  • Restrict file uploads to specific MIME types only
Top Performance Best Practices
  • Use $select in Liquid entity queries — fetch only needed columns
  • Add FetchXML top attribute — never fetch unbounded result sets
  • Cache expensive Liquid results using {% cache %} tag (object cache)
  • Minimize web files (combine CSS/JS where possible)
  • Use CDN for static assets — Power Pages supports Azure CDN integration
  • Use lazy loading for images on long pages
  • Avoid complex Liquid logic in header/footer templates (runs on every page)
Liquid Object Cache
Use {% cache 'key' timeout:300 %} ... {% endcache %} to cache rendered output for N seconds. Great for expensive FetchXML queries used on every page (e.g., navigation categories, product lists).
Portal Checker tool
Built-in diagnostic tool in Power Pages admin center. Checks for: misconfigured authentication, missing table permissions, site settings issues, performance anti-patterns. Run before go-live and after major changes.
Not running Portal Checker before go-live is a common mistake. It can catch security misconfigurations (like open table permissions) that would expose data to all users.
