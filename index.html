Dataverse Fundamentals

▼
What is Microsoft Dataverse?
A cloud-based, low-code data platform built on Azure that stores and manages business data used by Power Platform and Dynamics 365. It provides built-in security, logic, and integration capabilities.
Dataverse was previously called Common Data Service (CDS). Mention this transition to show awareness.
How is Dataverse different from SQL Server / SharePoint?
FeatureDataverseSQL ServerSharePoint
Built-in Security✅ Row/column-levelManual setupLimited
Business LogicPlugins, Business RulesStored ProcsWorkflows only
APIOData/Web APICustomREST
Power Platform native✅ First-classVia connectorVia connector
What are the key capabilities of Dataverse?
  • Tables: Structured data storage (standard, custom, virtual, elastic)
  • Relationships: 1:N, N:1, N:N
  • Security: Role-based, row-level, column-level
  • Logic: Business rules, calculated columns, rollup fields
  • Integration: Web API (OData), connectors, Dataflows
  • Auditing & Compliance: Built-in audit history
What is a Dataverse Environment?
An isolated container that holds data, apps, flows, and customizations. Each org gets a Default environment; additional environments (Sandbox, Production, Developer) can be created per business need.
Don't confuse "Environment" with "Organization" — in Dataverse context, each environment has exactly one database (org). This is a classic trap question.
What is the Dataverse Web API?
A RESTful API based on OData v4 protocol. Used to perform CRUD operations, execute functions/actions, and query data. Endpoint format: https://<orgurl>/api/data/v9.2/
GET /api/data/v9.2/accounts?$select=name,revenue&$filter=revenue gt 100000

Tables, Columns & Relationships

▼
Types of Tables in Dataverse
TypeDescription
StandardMost tables — full CRUD, owned by users/teams
ActivityTime-based records (Email, Task, Phone Call). Inherit from ActivityPointer.

ElasticLarge-scale, high-throughput; uses Cosmos DB underneath

Virtual tables are great for read-only integration scenarios — bring up CosmosDB or external REST as examples.
What is the difference between Owner and Organization ownership?
  • User/Team owned: Each record has an owner. Security model applies sharing/team-based access.
  • Organization owned: All records belong to the org; any user with table access can see all records. No row-level ownership.
Currency table = Organization owned (all users see all currencies). Account = User/Team owned.
You CANNOT change table ownership type after creation. This is a classic interview trap!

What is a Polymorphic Lookup?
A lookup column that can reference multiple table types. Called "Customer" (Account or Contact) or "Regarding" (any activity-enabled table). Internally uses the activityparty pattern.
Relationship Behaviors (Cascading)?

ParentalCascadeCascadeCascade

Parental behavior means child records follow parent's lifecycle. Use Restrict Delete when you must prevent orphan records from being deleted accidentally.

Calculated fields cannot be used in other calculated fields. Rollup fields have a 12-hour recalculation cycle by default — manual recalculation is possible via a button. Both are common traps.
What is a Global Option Set vs Local Option Set?
  • Global: Defined once, reusable across multiple tables/columns. Changes propagate everywhere.
  • Local: Defined on a specific column, not reusable.
Use Global when the same choice list (e.g., Industry, Status) appears on multiple tables.
Date/Time Modes — when to use which?
  • User Local: Converts to/from UTC based on user's timezone. Default. Good for meeting times.
  • Date Only: No time stored; no timezone conversion. Good for birthdays, contract dates.
  • Time Zone Independent: Stored as-is, no conversion. Good for shift schedules spanning timezones.
Date/Time mode CANNOT be changed after column creation. Mention this to show depth.

Security Model

▼
Layers of Dataverse Security
  1. Environment Security: Azure AD roles control who can access the environment
  2. Security Roles: Define table-level and field-level privileges
  3. Business Units: Hierarchical grouping of users; roles are scoped to BU
  4. Teams: Group users to share roles & records
  5. Record Sharing: Share individual records with specific users/teams
  6. Column Security Profiles: Column-level security to restrict field read/write
  7. Hierarchy Security: Manager can access reports' records
Security Role Privileges — what are they?
Each table in a security role has: Create, Read, Write, Delete, Append, Append To, Assign, Share privileges, each scoped at: None → User → Business Unit → Parent BU → Organization.
Append vs Append To: Append = the child record can be attached to a parent. Append To = the parent allows children to be attached to it. Both must be granted for a lookup to work.
Interviewers often ask about Append vs Append To. If a user can't attach a Contact to an Account, check BOTH Append on Contact AND Append To on Account.
What are the types of Teams in Dataverse?
Team TypeDescription
Owner TeamCan own records; members inherit team's roles
Access TeamAuto-created per record for ad-hoc sharing; cannot own records
AAD Group TeamBacked by Azure AD group; auto-syncs membership
What is Column-Level Security (Field-Level Security)?
Restricts read/write/create access at the individual column level using Column Security Profiles. Assigned to users or teams. Not all columns support this — only custom or specific system columns.
How does Record Sharing work?
An owner or sys admin can share a record with a specific user/team with selected privileges. Sharing is additive — you can't revoke privileges below what the security role already grants. Managed via PrincipalObjectAccess table internally.
Mention that sharing doesn't replace roles — it supplements them. If a role gives Read at BU level, sharing can extend to records outside BU but can't restrict the role's existing access.

Business Rules

▼
What are Business Rules in Dataverse?
No-code logic applied to a table form or record. Can perform: Set field value, show/hide field, set field requirement level, show error message, lock/unlock field, validate data.
Business Rule Scope Options
ScopeWhere it runs
Entity (Table)Server-side — runs on ALL operations: API, import, plugins
All FormsClient-side on all forms for that table
Specific FormClient-side on one specific form only
Business Rules with scope "Entity" can only SET field values and SHOW ERRORS. They CANNOT show/hide fields or change requirement level at entity scope — those are form-only actions. Classic interview trap!
BR: If "Lead Source" = "Web" then set "Rating" to "Warm" — works at Entity scope. If "Hide Phone field when Type = Email" — only works at Form scope.
Business Rules vs JavaScript — when to use which?
  • Business Rules: Simple conditions, no-code, managed in solution, runs server-side (entity scope)
  • JavaScript: Complex logic, dependent lookups, dynamic UI, multi-step operations
Prefer Business Rules for simple validations; use JS for complex interactions.

Dataverse Queries & APIs

▼
Ways to Query Dataverse
MethodUse Case
Web API (OData)External apps, JS, HTTP clients
FetchXMLComplex queries, aggregates, plugins, reports
LINQ (SDK).NET plugins/integrations using CRM SDK
QueryExpressionProgrammatic queries in .NET SDK
Dataverse Search (Relevance Search)Full-text search across multiple tables
What is FetchXML?
An XML-based query language native to Dataverse. Supports joins, aggregates, grouping, paging. Used in Advanced Find, reports, and SDK queries.
<fetch top="10">
  <entity name="account">
    <attribute name="name"/>
    <filter><condition attribute="statecode" operator="eq" value="0"/></filter>
  </entity>
</fetch>
Key OData Query Options
  • $select — limit columns returned
  • $filter — filter records
  • $orderby — sort results
  • $top — limit rows (max 5000 without paging)
  • $expand — include related table data
  • $count — return record count
Dataverse Web API has a 5000-record page limit. For large datasets use @odata.nextLink for pagination or FetchXML with paging cookie.
What is Dataverse Search (Relevance Search)?
Azure Cognitive Search-powered full-text search. Searches across multiple tables simultaneously, supports fuzzy matching, relevance ranking, and faceted filters. Must be enabled per environment and tables opted in.

Audit, Duplicate Detection & Data Management

▼
How does Auditing work in Dataverse?
  • Tracks Create, Update, Delete, and read access on records
  • Must be enabled at: 1) Environment level, 2) Table level, 3) Column level
  • Audit logs stored in the Audit table
  • Can be viewed in the Audit History subgrid on records or Audit Summary view
Auditing has a 3-level hierarchy — if you enable it at environment level only but not table level, no data is captured. All three levels must be on!
Duplicate Detection in Dataverse
  • Configure Duplicate Detection Rules per table with match conditions
  • Runs on Create/Update or via on-demand Duplicate Detection Jobs
  • Duplicate records show a warning but don't block save by default for users with override privilege
  • Results stored in Duplicate Record table
Data Import & Export Options
ToolUse Case
Import from Excel/CSVUI-based bulk import via wizard
DataflowsETL from external sources; Power Query based
Data Export ServiceReplicate Dataverse to Azure SQL (deprecated — use Synapse Link)
Azure Synapse LinkContinuous export to Azure Data Lake/Synapse for analytics
Configuration Migration ToolMove configuration/reference data between environments
What is the Recycle Bin in Dataverse?
Soft-delete feature (if enabled) that retains deleted records for 30 days. Records can be restored. Admins can configure retention period. Not enabled by default on all environments.

Environments & Solutions

▼
Types of Environments
TypePurpose
DefaultAuto-created per tenant; everyone gets access
ProductionLive business data; backup/restore available
SandboxDev/Test; can be reset; copy from production
DeveloperFree, single-user; for personal dev/testing
Trial30-day; auto-deleted
What is a Solution?
A container for customizations and extensions (tables, forms, flows, apps, plugins, etc.) used for ALM. Two types:
  • Unmanaged: Used in development; fully editable; exports as source
  • Managed: Used in test/production; locked; controlled upgrade path
Unmanaged solution components ARE part of the "Default Solution" — they live in the org. Managed solutions layer on top. If you delete a managed solution, its components are removed. If unmanaged customizations override a managed layer, unmanaged takes precedence — this is "solution layering."
What are Solution Layers and how do they work?
When multiple solutions modify the same component, they stack in layers. The topmost (active) layer wins. You can view layers in the "See solution layers" option. Unmanaged customizations always sit above managed layers.
Managed vs Unmanaged Properties
UnmanagedManaged
Editable in target✅ Yes❌ Depends on managed properties
Delete removes components❌ No✅ Yes
Used inDev envTest/Prod
What is a Publisher?
Defines the customization prefix (e.g., cr123_) used for all custom components in a solution. Best practice: create a custom publisher instead of using the default "customizations" publisher with prefix new_.
Always use a custom publisher with a meaningful prefix in enterprise projects. Mixing prefixes across teams creates messy ALM.

Model-Driven App Fundamentals

▼
What is a Model-Driven App?
A Power Platform app type where the UI is derived from the data model (Dataverse tables). The app is built around forms, views, dashboards, and business processes. Contrast with Canvas Apps which are pixel-perfect and data-source agnostic.
Model-Driven vs Canvas App
FeatureModel-DrivenCanvas
Data SourceDataverse only300+ connectors
UI DesignAuto-generated from data modelPixel-perfect, drag & drop
Business LogicPlugins, BRs, FlowsPower Fx formulas
NavigationSitemap-drivenScreens-based
Best ForComplex data mgmt, CRM/ERP styleTask-specific, mobile-first
Components of a Model-Driven App
  • Sitemap: Navigation structure (areas → groups → subareas)
  • Tables: Which tables are included
  • Forms: Data entry UI
  • Views: List/grid of records
  • Dashboards: Charts, lists, iframes
  • Business Process Flows: Guided stage-based processes
  • Charts: Visualizations on views
  • Command Bar: Action buttons
What is a Business Process Flow (BPF)?
A guided stage-based process displayed at the top of a form. Helps users through a multi-step process. Each stage has steps (fields to fill). Can span multiple tables. Stored as a table internally (processstage). Max 5 tables per BPF, max 30 stages.
Opportunity BPF: Qualify → Develop → Propose → Close. Each stage requires specific fields before moving forward.
BPFs can branch (conditional branching) allowing different paths based on data. This is an advanced feature worth mentioning.

Forms

▼
Types of Forms in Model-Driven Apps
Form TypePurpose
MainPrimary data entry/edit form; full feature support
Quick CreateLightweight form for fast record creation (enabled per table)
Quick ViewRead-only form embedded in another form to show related record data
CardCompact view for interactive dashboards (stream tiles)
Form Structure
Form → Tabs → Sections → Fields/Controls
Each tab can have multiple sections. Sections organize fields in 1–4 column layouts.
Form Order & Security Roles
Multiple forms can exist per table. Users see the highest-ranked form they have access to. Forms can be assigned to specific security roles to show the right form per user type.
If a user isn't seeing the expected form, check: 1) Form order, 2) Security role assignment on the form. This is a common support scenario.
What are Subgrids on Forms?
Embedded list controls on a form showing related records. Uses a View from the related table. Can be configured for display only or allow inline create/edit.
What is a Quick View Form?
A read-only form embedded within a Main form to display fields from a related (lookup) record without navigating away. Configure it via "Quick View Control" in the form editor.
On a Contact form, embed an Account Quick View Form to show the parent account's address and phone directly.
Header, Footer, and Body in Forms
  • Header: Shows key fields at the top (up to 4 fields). Always visible.
  • Body: Main content area with tabs and sections
  • Footer: Legacy area; largely deprecated in modern forms

Types of Views in Model-Driven Apps

PublicAvailable to all users; created by customizers
SystemDefault views (Active, Inactive, All Records); protected

What is the Quick Find View?
Defines which columns are searched when a user types in the search box on a view. Add columns to "Find Columns" to make them searchable. The columns displayed in results are the "View Columns."
Just adding a column to the Quick Find view's visible columns doesn't make it searchable — you must add it to "Find Columns" separately. Very common interview and real-world trap!
How to set a Default View?
In the View editor, right-click a public view and select "Set as Default View." The default view is what users see when they first navigate to a table's list.
Views can be used in: List pages, Subgrids, Lookups, Dashboards. Show depth by mentioning all use cases.

Ribbons & Command Bar

▼
What is the Command Bar?
The action buttons at the top of forms and views (formerly called "Ribbon"). In modern MDA, this is the Command Bar. Contains system commands (Save, New, Delete) and custom commands.
How to customize the Command Bar?
  • Modern way: Power Apps Command Designer (no-code, in-solution)
  • Legacy way: Ribbon Workbench (XrmToolBox) for complex scenarios
Can add buttons with: JavaScript actions, Power Automate flow triggers, or URL navigation.
What are Display/Enable Rules?
  • Display Rule: Controls visibility of a command button
  • Enable Rule: Controls if a button is clickable (greyed out vs visible)
Can use JavaScript functions, entity privilege checks, form state, or selection count as conditions.
Command Bar customizations in Power Apps Command Designer are managed in solutions as appaction components — much cleaner than old RibbonDiff XML.

Sitemap & Navigation

▼
Sitemap Structure
App → Area → Group → Subarea
Areas = top-level navigation (left rail in unified interface). Groups organize subareas. Subareas point to entities, dashboards, web resources, or URLs.
Sitemap Editor
Built into Power Apps maker portal (make.powerapps.com). Drag-and-drop interface. Subareas can be: Entity (table), Dashboard, Web Resource, URL.
App Modules — what are they?
Each Model-Driven App is an "App Module." App Modules define: which entities are included, sitemap, dashboards, forms, views, and BPFs surfaced. Multiple app modules can share the same data model but show different subsets.
A "Sales App" module shows Accounts, Opportunities, Leads. A "Customer Service App" module shows Cases, Contacts, Knowledge Articles — same Dataverse, different UX.
App Modules support security roles — only users with the right role see the app in their app list. This is a clean way to control access.

JavaScript & Client-Side Scripting

▼
Where is JS used in Model-Driven Apps?
  • Form Events: OnLoad, OnSave, OnChange (field-level)
  • Command Bar: Button click actions
  • Subgrid Events: OnLoad, OnRecordSelect
  • Business Process Flow: Stage events
Key Xrm.Page / formContext APIs
APIPurpose
formContext.data.entity.getId()Get current record GUID
formContext.getAttribute("name").getValue()Get field value
formContext.getAttribute("name").setValue(val)Set field value
formContext.getControl("name").setVisible(bool)Show/hide field
formContext.getControl("name").setDisabled(bool)Lock/unlock field
formContext.getAttribute("name").setRequiredLevel("required")Make required
Xrm.Navigation.openForm()Navigate to record
Xrm.WebApi.retrieveMultipleRecords()Query Dataverse from JS
Xrm.Page is deprecated — always use executionContext.getFormContext() passed as a parameter. Interviewers check if you know this distinction!
How do you register a JS event handler?
  1. Create a JS Web Resource
  2. Add it to the form via Form Properties → Events
  3. Register function on the desired event (OnLoad, OnSave, OnChange)
  4. Pass execution context as first parameter (check "Pass execution context as first parameter")
Async JS on OnSave — how to prevent premature save?
Use saveContext.getEventArgs().preventDefaultOnError() or return a Promise from OnSave handler. Modern approach: async OnSave handlers are supported — return a Promise and Dataverse awaits resolution before saving.
Always use preventDefault() in OnSave if you need to cancel save after validation. Use async/await pattern with Xrm.WebApi for cleaner code.

Plugins

▼
What is a Plugin?
A .NET class library registered to execute on Dataverse platform events (CRUD, Associate, etc.). Runs synchronously in the platform pipeline. Used for complex server-side business logic.
Plugin Pipeline Stages
StageNumberDescription
Pre-Validation10Before security checks; outside transaction; DB reads OK, no rollback
Pre-Operation20After security checks; inside transaction; can modify InputParameters
Main Operation30Core platform op (not available for custom plugins)
Post-Operation40After op completes; inside transaction; can modify OutputParameters
Pre-Validation runs OUTSIDE the database transaction. If you do DB writes there and an error occurs in Pre-Operation, those writes won't roll back! Use Pre-Operation for anything that must be atomic.
Synchronous vs Asynchronous Plugins
  • Sync: Blocks the operation; runs in-transaction; immediate; 2-minute timeout
  • Async: Queued; runs in background via Async Service; no timeout concern; can't modify operation
Use Async for time-consuming operations (email sends, external API calls).
Key Plugin Context Members
MemberPurpose
InputParameters["Target"]The record being operated on (Entity type)
OutputParameters["BusinessEntityCollection"]Query results
PreEntityImagesRecord state BEFORE the operation
PostEntityImagesRecord state AFTER the operation
InitiatingUserIdUser who triggered the action
DepthPlugin call depth (avoid infinite loops)
Always check context.Depth in plugins to prevent infinite loops when a plugin triggers the same message recursively.
What are Entity Images in Plugins?
Snapshots of the record. Pre-Image = state before operation (must be registered). Post-Image = state after operation. Pre-Image not available in Pre-Validation. Post-Image not available in Pre-Operation.
Plugin Registration
Done via Plugin Registration Tool (from PAC CLI or XrmToolBox). Register: Assembly → Plugin Type → Step (message, stage, mode, filtering attributes) → Images.
If you don't specify Filtering Attributes on a plugin step, the plugin fires on EVERY update regardless of which field changed. Always filter to relevant attributes to avoid unnecessary execution!

Power Automate Integration

▼
Dataverse triggers in Power Automate
TriggerNotes
When a row is added, modified or deletedStandard CRUD trigger
When a row is selected (V2)Triggered from command bar button in MDA
When an action is performedCustom API / Action trigger
RecurrenceScheduled flows
Classic Workflow vs Power Automate (Cloud Flow)
Classic Workflows are deprecated but still functional. Power Automate Cloud Flows are the recommended replacement. Cloud Flows are more powerful, support parallel branches, HTTP calls, and 400+ connectors.
What is a Custom API (Custom Action)?
A reusable, callable operation defined in Dataverse. Can be triggered from: plugins, Power Automate, JavaScript (Xrm.WebApi), or external HTTP. Supports input/output parameters. Replaces old "Custom Workflow Activities" for complex reusable logic.
Custom APIs are the modern replacement for Custom Actions defined in the legacy process designer. They're solution-aware and support bound/unbound patterns just like standard Web API functions.

Power Apps Component Framework (PCF)

▼
What is PCF?
Power Apps Component Framework lets developers build custom UI controls using TypeScript/React. PCF controls replace default field controls or dataset views with rich, interactive custom UIs.
PCF Control Types
  • Field Control: Replaces a single field's UI (e.g., a star rating input instead of a number)
  • Dataset Control: Replaces a view/subgrid with custom visualization (e.g., Kanban board, calendar)
PCF Lifecycle Methods
  • init(): Initialize component, subscribe to events
  • updateView(): Called when data changes; refresh UI
  • getOutputs(): Return current output values
  • destroy(): Cleanup on removal
Build a PCF "Color Picker" control for a text field that stores hex color codes — shows a color wheel instead of a text box.
PCF controls are packaged as solutions. Virtual controls (no manifest input output) are the modern approach. React Virtual controls give full React rendering without iframe overhead.

ALM & Deployment

▼
ALM Best Practices in Power Platform
  1. Develop in a Developer/Sandbox environment
  2. Use source control (Git) with PAC CLI to extract solution files
  3. Export as Unmanaged from Dev; build Managed for deployment
  4. Deploy Managed to Test → Production
  5. Use Pipelines (Power Platform Pipelines) or Azure DevOps for CI/CD
PAC CLI — key commands
CommandPurpose
pac auth createConnect to environment
pac solution exportExport solution
pac solution importImport solution
pac solution unpackUnpack .zip to source files
pac solution packPack source files to .zip
pac pcf init/pushPCF development
What are Managed Properties?
Settings on solution components that control what customers/partners can customize in managed solutions. E.g., "Can be customized" = false means a field cannot be modified when installed as managed.
Solution Dependencies
Solutions track dependencies between components. You can't delete a component if another component depends on it. Use "Show Dependencies" to troubleshoot. Missing dependencies cause import failures in target environments.
Importing a managed solution that has a missing dependency (e.g., references a table from another solution not installed) will FAIL with a dependency error. Always deploy base solutions first!

Performance & Best Practices

▼
Top Performance Best Practices for Dataverse
  • Use $select to return only needed columns (avoid over-fetching)
  • Add alternate keys or indexes on frequently queried fields
  • Avoid retrieving more than 5000 records at once — use pagination
  • Use ExecuteMultiple for bulk operations (reduces API round trips)
  • Avoid deep plugin chains and long synchronous plugin execution
  • Use RetrieveMultiple with column sets instead of full entity
  • Cache reference data in plugins using static/lazy-loaded patterns
Top Best Practices for Model-Driven Apps
  • Keep OnLoad JS lightweight — defer heavy logic or use async
  • Minimize Web API calls in JS — batch where possible
  • Use Business Rules over JS for simple logic (less maintenance)
  • Configure form-level filtering attributes to prevent unnecessary plugin triggers
  • Use Quick Find view with appropriate find columns (not every column)
  • Use App Modules to show only relevant tables per user role
  • Avoid too many subgrids on one form — impacts load time
API Rate Limits & Throttling
Dataverse enforces service protection limits: 1000 requests per 5 minutes per user. Responses include Retry-After header when throttled. Use batching (OData $batch or ExecuteMultiple) to stay within limits.
What is the 2-minute plugin timeout?
Synchronous plugins must complete within 2 minutes. If exceeded, the platform terminates the plugin and rolls back the transaction. Move long-running logic to Async plugins or Power Automate.
For high-volume scenarios, consider Elastic Tables (Cosmos DB backed) for write-heavy telemetry/audit data — they offer horizontal scaling beyond what standard Dataverse tables provide.
Common Solution / ALM Mistakes to Avoid
  • Making changes directly in Production (never do this)
  • Using "Default Solution" as your working solution — always use a custom publisher
  • Not specifying filtering attributes on plugins
  • Exporting managed solutions from Dev (should be unmanaged from Dev)
  • Not handling solution dependency order in deployments
